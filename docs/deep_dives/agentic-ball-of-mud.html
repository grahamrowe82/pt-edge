Most agentic AI systems today have the same fundamental architecture as a badly written cron job: a single ephemeral process, holding all state in local memory, executing steps sequentially, with no durable record of what it did or why.

The context window is the "memory." If it gets truncated, knowledge is lost. The LLM's next-token prediction is the "orchestrator." If it hallucinates a step, there's no checkpoint to roll back to. If the process crashes, you start from scratch. There's no idempotency, no separation of concerns, no observability beyond whatever the agent happens to print to stdout.

This is not a new failure mode. Backend engineering identified and solved every one of these problems over the last two decades. The agent ecosystem is rediscovering them — slowly, out of order, and mostly by reinventing existing infrastructure badly.

Problem 4: Crash recovery — the void

If step 7 of 12 fails, you rerun 1–12. The fix in backend engineering: make each step atomic, record completion, resume from where you left off. In the agent ecosystem, this problem is almost completely unaddressed.

What good looks like

The projects that actually work well tend to rediscover backend patterns by accident. Using git as the source of truth. Writing intermediate results to files or databases. Structuring work as discrete tasks rather than one long chain of thought. They're reinventing the work queue — they just don't call it that because the marketing says "agent."

The bridge projects are the ones to watch:

  • trigger.dev — background jobs infrastructure being adopted by agent developers ({{ '{:,}'.format(767768) }} downloads/month)
  • Sayiir — "simplified Temporal" in Rust, explicitly targeting AI agent workflows
  • Stabilize — queue-based state machine sitting at exactly the right abstraction level
  • DBOS + LlamaIndex — the first integration between a durable execution runtime and an agent framework
  • AxmeAI — pre-launch company building durable execution "where agents, services, and humans coordinate as equals"

The interesting research question: is there something inherent in the LLM-as-orchestrator pattern that resists these fixes? Can you have a durable, observable, crash-recoverable agent that still benefits from LLM reasoning at each step? Probably yes — but it looks much more like a worker pulling tasks from a queue than an "autonomous agent" with a system prompt and a dream.

The signal from the pain

The industry is feeling this. Amazon's Kiro agent deleted and recreated environments, causing a 13-hour AWS outage (Pragmatic Engineer, March 2026). Latent Space reports agents now run 6–8 hour tasks. Anthropic published guidance on "harness engineering for long-running tasks." On Hacker News, a developer asks "How do you cope with the broken rhythm of agentic coding?" while another builds "Agent Kernel — Three Markdown files that make any AI agent stateful."

Sentrial (YC W26) raised money specifically to "catch AI agent failures before your users do." When Y Combinator is funding the observability gap, it's real.

The agent ecosystem is solving these problems in the wrong order — from the easiest (memory) to the hardest (crash recovery) — and the projects that will define the next generation of agent infrastructure are the ones that bring proven backend engineering patterns into the LLM world. Not smarter orchestration within the process. The fix is killing the process as the locus of truth and putting the truth somewhere that survives it.
