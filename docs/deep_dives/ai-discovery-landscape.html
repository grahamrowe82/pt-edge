The awesome list problem

The registry problem

Registries are the second generation. Instead of a markdown file maintained by one person, they are structured databases with search, categories, and (sometimes) automated ingestion. In the MCP ecosystem, at least five competing registries index MCP servers: the official Anthropic registry, Smithery, MCP.so, PulseMCP, and SkillsIndex.

Our data on the MCP registry aggregator category: {{ cats.get('mcp:mcp-registry-aggregators', {}).repo_count | default(67) }} repos, {{ cats.get('mcp:mcp-registry-aggregators', {}).total_stars | intcomma }} combined stars. The quality gap score is just {{ cats.get('mcp:mcp-registry-aggregators', {}).quality_gap_score | default(17) }}/100 — meaning the demand for MCP discovery is relatively well-served compared to other categories.

But every single one of these registries is a flat catalogue. Search by name, maybe filter by category, read a description someone wrote on day one. None of them answer the question that actually matters: should I use this?

This is not a quality problem — it is a design problem. Registries optimise for completeness (list everything) not for decision-making (recommend the best option). When an AI agent searches for "a Postgres MCP server," it doesn't need a list of 47 options. It needs the three that are well-maintained, have package distributions, and won't break.

What breaks at scale

The AI open-source ecosystem adds thousands of repos per month. We track {{ global_total | default('226,000+') }} across 18 domains. No human can curate that. The specific things that break:

  • Staleness: A list of "best Python ML libraries" that hasn't been updated in 6 months is actively harmful — it recommends abandoned projects and misses new ones.
  • No quality signals: Awesome lists sort by name or category. They can't tell you that a project's last commit was 2 years ago, or that it has no license, or that nobody actually downloads it.
  • No cross-vertical view: awesome-langchain covers LangChain. awesome-mcp-servers covers MCP. Nobody covers "what's the best way to build a document Q&A pipeline" which requires cross-cutting knowledge across RAG, embeddings, vector databases, and LLM tools.
  • No negative signals: Awesome lists only add. They never warn you away from something. There's no "this project was abandoned" or "this has known security issues" or "this scores 15/100 on quality."

The third generation: programmatic intelligence

The answer is to stop curating by hand and start computing from data. Track every repo programmatically. Score them daily on objective signals. Cluster them into categories using embeddings, not human judgement. Surface the results through both human-readable pages and machine-readable APIs.

What's still missing

Programmatic scoring solves the maintenance and scale problems but creates new ones:

  • Editorial judgement: Data can tell you that a project scores 72/100. It can't tell you that it's the best choice for your specific use case. That's what deep dives like this one are for — combining live data with human analysis.
  • Recommendation context: "Best overall" is not always "best for you." A Python developer and a Rust developer need different recommendations from the same category. Personalisation is the next layer.
  • Real-time signals: Daily scoring catches trends over days and weeks. It doesn't catch a critical security vulnerability disclosed this morning. Integration with security advisory feeds, CVE databases, and real-time social signals is the frontier.

The ecosystem is moving from curated lists to computed intelligence. The question is not whether the awesome list format will be replaced — the graveyard count proves it can't scale. The question is what the replacement looks like, and whether it serves both human developers and the AI agents that are increasingly making tool selection decisions on their behalf.
