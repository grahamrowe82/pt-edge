You ask your AI coding assistant "I need to add voice to my app." It recommends a TTS library. You integrate it. It works. Then you realise you also need speech-to-text. You ask again, integrate again. Then you discover your voice agent can't understand intent. Then you learn about latency budgets. Then you find out you have no way to measure whether any of it is actually working.

Each time, the AI cheerfully helps you solve the piece you asked about. It never volunteers that you're working at the wrong level of abstraction — that "adding voice to your app" is actually a five-layer architecture problem.

Here's the full stack.

Layer 1: Speech-to-text — just use Whisper

This is a solved problem. The only decision is which Whisper variant: WhisperX (83/100) for Python with speaker diarisation, faster-whisper (68/100) for speed, whisper.cpp for C++ edge deployment, sherpa-onnx (84/100) for mobile and embedded. Don't agonise. Pick one. Move on.

Layer 2: Text-to-speech — this is where you actually have a decision

Best quality, paid: ElevenLabs (85/100). Industry benchmark.

Free and good-enough: edge-tts (69/100, 10K+ stars). Wraps Microsoft Edge's TTS API — high quality, zero cost, unofficial API (risk: Microsoft could cut it off).

Local inference: mlx-audio (86/100) for Apple Silicon. sherpa-onnx for everything else.

Real-time streaming: RealtimeTTS (77/100). Streams audio as it's generated — critical for voice agents where 200ms to first audio is the difference between natural and broken.

Layer 3: Intent and understanding — the layer AI never mentions

Between "the user said something" and "the app responds," you need to understand what they meant. For commands, keyword matching works. For conversation, you need either embeddings (semantic similarity against known intents) or an LLM (open-ended understanding).

The thing nobody warns you about: the latency budget. STT (200-500ms) + LLM (500-2000ms) + TTS (200-500ms) = 1-3 seconds of silence. A chatbot can take 3 seconds. A voice assistant that takes 3 seconds sounds broken. This is why local inference and streaming TTS matter — they compress the edges so the LLM has more time in the middle.

Layer 4: Orchestration — assembling it yourself

If you're building a voice agent, there's no "Rails for voice agents" yet. You're wiring STT → understanding → LLM → TTS yourself, handling interruption, managing conversation state, optimising latency. rapida voice-ai (69/100) and voice-devtools are building toward this, but the space is early.

Layer 5: Evaluation — nobody thinks about this until production

How do you know your voice app works for real users? Most evaluation tools focus on ASR word error rate. Almost nothing exists for TTS quality, conversation quality, or end-to-end latency in production. autovoiceevals is one of the few attempts. Plan to build custom metrics.

The full picture

An AI coding assistant would walk you through layer 1, then layer 2 if you asked, and never mention layers 3-5 until you hit a wall. Now you know the full landscape before you start.
