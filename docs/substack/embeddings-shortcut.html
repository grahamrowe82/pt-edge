Every week, developers build keyword search systems, tune Elasticsearch clusters, or write regex pipelines to match text. And every week, they could have replaced the whole thing with an embedding API call and a Postgres query.

Embeddings sound like ML research. They're not. In 2026, generating an embedding is an API call that costs fractions of a cent. Storing and searching embeddings is a Postgres extension. The entire workflow is 20 lines of code and the results are dramatically better than keyword matching.

What embeddings actually are

An embedding turns text into a list of numbers. Similar text produces similar numbers. "Government infrastructure tender" and "public works construction bid" have near-identical embeddings despite sharing zero keywords. That's the magic: semantic similarity without keyword matching.

You generate one with a single API call. It costs less than $0.00002. No model training, no GPU, no data pipeline.

Just use Postgres

The vector database market wants you to believe you need specialised infrastructure. For most applications, you don't.

pgvector is a Postgres extension. If you already run Postgres, you add the extension, create a vector column, and query with ORDER BY embedding <=> query_vector LIMIT 10. No new database, no new vendor.

The threshold is roughly 1 million vectors. Below that, pgvector is simpler, cheaper, and fast enough. Above that, the dedicated vector databases earn their keep: Qdrant (87/100), Chroma (87/100), Weaviate (87/100), LanceDB (87/100). All excellent, all overkill for most use cases.

Which embedding model?

If you have an OpenAI key: text-embedding-3-small. Cheap, good quality, no infrastructure. Done.

If you want local: fastembed (77/100) from Qdrant — runs on CPU. mlx-embeddings (69/100) for Apple Silicon.

Best open-source: FlagEmbedding (82/100) — the BGE models that top the MTEB benchmark (92/100).

What embeddings replace

This is the part nobody says clearly enough. Embeddings don't supplement your existing approach. They replace it:

  • Keyword search: Embeddings find semantically similar results when keywords differ. No synonym dictionaries, no stemming.
  • Regex matching: Embed your categories, embed the input, take the closest match. Works across languages.
  • Elasticsearch: The operational cost of a cluster usually exceeds what you get from it. pgvector gives better relevance with zero additional infrastructure.

The cost reality

Embedding 100,000 documents: ~$2. Each search query: ~$0.00002. Storage for 1M vectors: ~6GB in standard Postgres. Compare to running an Elasticsearch cluster ($50-500/month) or the engineering hours writing regex.

Getting started in 15 minutes

  1. Get an embedding API key (OpenAI, Cohere, or install fastembed for local)
  2. Add pgvector: CREATE EXTENSION vector;
  3. Create a column: ALTER TABLE docs ADD COLUMN embedding vector(1536);
  4. Embed your data (loop, API call, store)
  5. Search: SELECT * FROM docs ORDER BY embedding <=> $query LIMIT 10;

No ML pipeline, no model training, no GPU. If you've been putting off "adding AI" because it seemed complex, this is the shortcut.
