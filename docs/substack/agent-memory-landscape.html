The context window paradox

Start with the assumption most builders make: "I have a 1M token context window, so I don't need a memory system." The data says otherwise. GPT-5.4 accuracy degrades to 36% at 512K-1M tokens. Context windows have plateaued at 1M for two years, constrained by hardware limits. Even within that 1M, practical performance drops off a cliff above 256K tokens.

This is why 18 newsletter articles from Latent Space, Zvi, Simon Willison, and One Useful Thing all covered memory and context in the last 90 days. The consensus: persistent memory systems are becoming the way to give agents long-term state, because context windows alone can't do it reliably.

Four projects defining the category

mem0 (49,600+ stars, Apache-2.0) is the category definer. Multi-level memory covering user, session, and agent state, with Python and JavaScript SDKs plus LangGraph and CrewAI integrations. Claims 26% higher accuracy with 90% lower token usage versus full-context approaches. If you're starting from zero, this is where you begin.

claude-mem (34,400+ stars) is the Claude Code ecosystem play. Captures everything Claude does, compresses it with AI, and provides progressive disclosure — surfacing relevant memories without flooding the context window. Eight releases in the last 30 days.

Cognee (13,200+ stars, 393 commits in 30 days) takes a different approach: graph plus vector hybrid retrieval with ontology grounding. If your use case requires understanding relationships between memories — not just retrieving similar ones — this is the most mature option. Highest development velocity in the entire category.

Memvid (13,400+ stars) is the radical simplicity play. Stores memories in a single .mv2 file with sub-5ms retrieval. No database server, no network calls, no infrastructure. Written in Rust.

The storage decision

The most fundamental architectural choice. Four approaches, each with real trade-offs:

Vector DB (Qdrant, Chroma, Milvus, LanceDB) — best for semantic similarity retrieval. The established approach, well-tooled, but struggles with relationships between memories and temporal reasoning.

Graph DB (Cognee, Graphiti at 23K+ stars) — best when memories have structure and relationships matter. Knowledge graphs capture "A depends on B" in ways vectors can't. More complex to operate.

SQL-native (Memori at 12,300+ stars) — the pragmatic middle ground. Intercepts LLM conversations automatically, stores in SQL, achieves 81.95% accuracy at roughly 5% of the token cost of full context. Use what you already know.

File-based (Memvid, ReMe) — the "you don't need a vector database" counter-narrative gaining traction on Hacker News. Human-readable Markdown persistence, zero infrastructure dependencies. Best for coding agents and single-user scenarios.

Where this is heading

The big question: will platform-native memory features (Claude Memory Export, Microsoft Copilot Tasks persistent memory) absorb this entire category? History suggests they'll serve the common case but leave room for specialized systems. Platforms provide simple key-value memory. Production agents need episodic recall, semantic compression, procedural learning, and temporal reasoning. That gap is where these 977 repos live.
