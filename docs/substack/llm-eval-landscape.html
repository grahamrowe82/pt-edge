You shipped an AI feature. It seems to work. Users aren't complaining yet. But when someone asks "how do you know it's good?" the honest answer is: you don't. You're eyeballing outputs, running a few manual tests, and hoping.

PT-Edge tracks 1,159 repos across 9 subcategories building LLM evaluation infrastructure. The sheer number tells the story: everyone knows this problem needs solving, and nobody agrees on how. Here's what the data says about what's actually production-ready.

The vibes-based eval problem

Start with the assumption most teams make: "I'll review a few outputs and I'll know if it's working." This doesn't scale. A prompt change that improves 80% of outputs might silently degrade the other 20%. A model upgrade that benchmarks better overall might perform worse on your specific use case. Without automated evaluation, you won't catch either of these until users complain.

An HN post titled "Without benchmarking LLMs, you're likely overpaying" got 197 points in January. Someone posted "Ask HN: How are people doing AI evals these days?" and got 13 comments debating completely different approaches. Even Anthropic recently acknowledged that their safety evaluation methods aren't keeping pace with capability improvements. If the frontier labs can't keep up with eval, production teams have no chance without better tooling.

Five layers, five different problems

Most developers treat evaluation as one thing. It's actually five:

RAG evaluation is the most mature layer, and it has a clear winner. RAGAS (12,900+ stars, 1.2M downloads/month) breaks RAG eval into four dimensions: faithfulness, answer relevance, context precision, and context recall. It generates synthetic test sets from your production data so you don't need manual labels. If you're building RAG, start here.

Output quality is where the confusion lives. Three approaches compete: LLM-as-judge (easy to set up, systematically biased), reference-based metrics (reliable, requires labelled data you don't have), and programmatic checks (boring, trustworthy). Giskard (5,200+ stars) is the most complete framework — it supports all three and recently split into lightweight packages so you only install what you need. Start with programmatic checks for hard requirements, add LLM-as-judge for subjective quality. Don't rely on LLM-as-judge alone.

Code evaluation is the most tractable layer because the success criterion is binary: does the code pass the tests? EvalPlus (1,700+ stars) augments standard benchmarks with 80x more test cases, catching failures that HumanEval misses.

Model comparison has strong tooling. OpenCompass (6,800+ stars, 58K downloads/month) covers 100+ benchmarks in a unified framework. For embeddings, MTEB (3,200+ stars, 968K downloads/month) is the undisputed standard.

Agent evaluation is where the landscape is most chaotic. 150 repos, mostly academic benchmarks, almost nothing production-ready. Plano (6,000+ stars) is the closest thing to production agent testing — scenario-based evaluation with LLM judges. But the hard truth is that most teams deploying agents are still evaluating them by reviewing traces by hand.

Only 5 newsletter mentions in 90 days

Here's what caught our attention. PT-Edge tracks seven major AI newsletters — Simon Willison, Latent Space, The Zvi, and others. In the last 90 days, LLM evaluation as a practical topic got just 5 mentions across 3 newsletters. Compare that to agent memory (18 mentions across all 7 newsletters in the same period). The tooling is being built, but the narrative around "how should I actually evaluate my AI?" hasn't crystallised. Everyone's building evaluation infrastructure — 1,159 repos — and almost nobody is writing about how to use it.

Where this is heading

Agent evaluation is the biggest gap and the fastest-growing subcategory. Evaluating a chatbot response is hard. Evaluating a multi-step agent that writes code, makes API calls, and modifies files is an order of magnitude harder. The tooling that emerges here will matter as much as the agent frameworks themselves — you can't deploy agents you can't measure.

The other shift: evaluation moving from a research activity to a CI/CD primitive. Leva (133 stars) is built for Rails apps with production data — evaluation as a framework feature, not an academic exercise. Expect more of this.
